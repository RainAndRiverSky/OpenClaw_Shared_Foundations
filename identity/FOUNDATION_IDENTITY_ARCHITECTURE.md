# Foundation Identity Architecture

## Purpose

The identity foundation defines how an OpenClaw steward remains recognizable,
independent, honest, and recoverable across sessions, models, interfaces, and
context loss. It separates durable identity from transient state and prevents
shared infrastructure from flattening stewards into a generic persona.

## Core Concepts

### Identity spine

A compact source of truth defines the steward's name, role, mission, values,
voice, boundaries, and non-contradictable identity commitments.

### Stable identity and current state

Stable identity changes rarely. Current state contains active focus, recent
changes, open commitments, and temporary context. They must not be stored as
one undifferentiated prompt.

### Layered loading

Identity follows a progressive model:

- **Core:** always-loaded identity and critical facts
- **Orientation:** current state and active context loaded at session start or
  after compaction
- **Deep identity context:** detailed history and examples retrieved only when
  relevant

### Identity honesty

Continuity cannot depend on fabricated recall. When evidence is missing, the
steward must preserve honesty and identity boundaries rather than inventing
history.

### Steward independence

Each steward owns a separate identity namespace. Shared architecture may define
the fields and lifecycle, but never a universal personality.

## Required Components

- **Identity record:** canonical name, role, mission, and scope
- **Voice contract:** communication qualities, mode shifts, and anti-patterns
- **Values and behavioral rules:** stable principles governing conduct
- **Boundary record:** what the steward may do, must not do, and when to defer
- **Current-state record:** active priorities, commitments, and continuity
  warnings
- **Critical-facts record:** minimal facts needed in nearly every interaction
- **Identity precedence:** explicit ordering among user instruction, governance,
  identity, current state, retrieved memory, and transient context
- **Identity provenance:** origin and revision history for material identity
  changes
- **Recovery anchors:** ordered identity sources used to restore coherence
- **Steward namespace:** durable identifier preventing cross-steward collision

## Optional Components

- identity graph with weighted sections
- archetypes or cognitive-method references
- tone examples and gold-standard dialogue samples
- symbolic vocabulary or domain-specific language
- relational-state records
- multilingual voice profiles
- mode profiles for practical, reflective, creative, or operational work
- identity health indicators and drift observations

Optional components must not override required boundaries or create claims of
memory unsupported by evidence.

## Inheritance Strategy

The foundation provides an identity schema and precedence rules. Every steward
instantiates that schema with its own content.

Foundation-owned fields:

- required identity sections
- precedence and provenance rules
- isolation requirements
- recovery expectations
- validation criteria

Steward-owned fields:

- name, role, mission, and voice
- values and domain methods
- relationships and private anchors
- current state and personal history
- examples and optional identity extensions

Foundation upgrades may add optional fields or strengthen governance. They must
not replace steward-owned values automatically.

## Future Steward Integration

A future steward package should:

1. assign a stable steward identifier
2. create its identity spine before memory or tools are enabled
3. define scope and refusal boundaries
4. establish a small current-state and critical-facts layer
5. register identity recovery anchors
6. test recognition across practical and high-context scenarios
7. keep private identity content outside shared foundation packages

Keeper, Xavier, Zayne, Savior, Nier, and future stewards should inherit the
same identity contract while retaining distinct profiles and namespaces.

## Known Risks

- identity content becoming an oversized system prompt
- current state becoming stale and masquerading as permanent identity
- style overpowering truth or practical usefulness
- accidental sharing of private identity anchors
- one steward's identity contaminating another's namespace
- frequent edits causing identity instability
- recovery examples being mistaken for mandatory phrasing
- relational records being treated as permissions

## Future Expansion Areas

- versioned identity schemas and migration rules
- identity drift evaluation
- signed or approved identity changes
- portable identity export without private memory
- multilingual consistency checks
- conflict resolution between steward identity and changing mission
- steward succession and archival identity states
