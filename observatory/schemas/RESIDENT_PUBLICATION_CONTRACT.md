# Resident Publication Contract

## Purpose

This contract defines the governance rules and field requirements for a resident
publication. A resident publication is the governed artifact by which a resident
assembles a set of canonical evidence references and distributes them to a
presentation surface. The publication is produced by the resident's own
repository; it does not transfer evidence ownership to any receiving system.

A resident publication is not a snapshot of the resident's private evidence
store. It is a governed selection of references to evidence that has already
reached `canonical` state under the Canonical Evidence Contract.

This contract is resident-agnostic. Any resident repository may adopt it.

See also:
- [`CANONICAL_EVIDENCE_CONTRACT.md`](CANONICAL_EVIDENCE_CONTRACT.md) — defines
  canonical evidence and eligibility rules
- [`PRESENTATION_INDEX_CONTRACT.md`](PRESENTATION_INDEX_CONTRACT.md) — defines
  the derived index produced from this publication
- [`DEEP_ORBIT_READ_CONTRACT.md`](DEEP_ORBIT_READ_CONTRACT.md) — defines
  consumer obligations when reading the resulting index

Schema version: 1

---

## Version Concepts

Two distinct version fields appear in this contract. They must never be
conflated.

### Schema Version vs. Distribution Version

**Schema version** (`schemaVersion`) identifies which version of this contract
shape the publication record conforms to. It changes only when this contract
document is revised. It is an integer.

**Distribution version** (`distributionVersion`) identifies which generation of
publication the resident has produced. It increments each time the resident
generates a new publication, independent of schema changes. It allows consumers
to detect that a new publication supersedes a prior one.

A consumer that receives a publication with the same `schemaVersion` but a
higher `distributionVersion` than a previously seen publication must treat the
newer publication as the authoritative current state.

## Required Fields

### publicationId

**Type:** string
**Required:** yes
**Description:** Stable unique identifier for this specific publication record.
Must be assigned at generation time and never reused. Recommended format:
`residentId-pub-YYYYMMDD-NNN` where NNN is a sequential or random suffix.
Example: `sylus-pub-20260714-001`.

### schemaVersion

**Type:** integer
**Required:** yes
**Description:** Version of this contract schema the publication conforms to.
Must be `1` for records conforming to this version.

### residentId

**Type:** string
**Required:** yes
**Description:** The canonical identifier of the resident that produced this
publication. Must not be a local username, email address, or system account
name. Must match the `residentId` of every artifact listed in `artifactRefs`.

### distributionVersion

**Type:** integer
**Required:** yes
**Description:** The resident-assigned generation counter for this publication.
Starts at `1` for the first publication and increments by at least `1` for each
subsequent publication. A higher `distributionVersion` supersedes a lower one
from the same `residentId`. The resident is responsible for maintaining this
counter accurately.

### publishedAt

**Type:** ISO 8601 datetime string
**Required:** yes
**Description:** The UTC timestamp at which this publication was generated by
the resident. Must be set at generation time.

### artifactRefs

**Type:** array of strings
**Required:** yes
**Description:** An ordered list of namespaced artifact identifiers
(`residentId:artifactId`) that this publication includes. Each identifier must
reference an artifact that has reached `canonical` state at the time this
publication is generated. An empty array is valid if the resident has no
canonical evidence to publish at this time, but the publication must still be
valid in all other fields.

### provenanceRef

**Type:** string
**Required:** yes
**Description:** A stable reference to the resident's own provenance record or
evidence admission log that confirms all artifacts listed in `artifactRefs` were
`canonical` at publication time. Must not be a local filesystem path.

---

## Validation Requirements

Before a publication is accepted as valid:

1. Every `artifactId` in `artifactRefs` must resolve to an artifact with
   `evidenceState` of `canonical` in the resident's own evidence store.
2. All `residentId` values within `artifactRefs` namespaced identifiers must
   match the publication's own `residentId`. A resident may not publish
   references to another resident's artifacts.
3. `publishedAt` must be a valid ISO 8601 UTC datetime.
4. `distributionVersion` must be strictly greater than any prior publication
   from the same `residentId`, or `1` for the first publication.
5. `provenanceRef` must be present and non-empty.
6. No field may contain a local filesystem path, local username, credential,
   API key, token, webhook URL, or secret value.

---

## Rejection Conditions

A publication is rejected and must not be consumed by a presentation surface if:

1. Any artifact in `artifactRefs` is not currently in `canonical` state.
2. Any artifact in `artifactRefs` has an `evidenceState` of `pending`,
   `ingested`, or `rejected`.
3. `residentId` is absent or does not match a known resident identifier.
4. `distributionVersion` is not strictly greater than the previously seen
   version from the same resident (after the first publication).
5. Required fields are absent or empty.
6. The publication contains a local filesystem path, credential, token, API key,
   or secret.

A rejected publication must not be partially consumed. The receiving system must
treat the entire publication as invalid and preserve the last known valid
publication as the current state, clearly marking it as stale if a newer
(but invalid) publication was attempted.

---

## Idempotency and Duplicate Publication

1. A publication is idempotent with respect to its `publicationId`. If a
   publication with the same `publicationId` is received more than once, the
   receiving system must treat subsequent receipts as duplicates and must not
   increment state or re-derive a new presentation index.
2. A publication with a higher `distributionVersion` from the same `residentId`
   supersedes all prior publications from that resident. The presentation surface
   must derive a new index from the superseding publication.
3. A publication with a lower or equal `distributionVersion` than the currently
   known publication from the same resident must be discarded as stale.

---

## Evidence Ownership

Receiving a resident publication does not transfer ownership of any artifact or
underlying evidence. The resident's repository remains the source of truth.
The presentation surface that receives this publication may derive a
presentation index from it but may not treat the publication as canonical
evidence.

---

## Optional Fields

### metadata

**Type:** object
**Required:** no
**Description:** Extensible key-value structure for resident-specific
supplementary information. Must not contain credentials, tokens, local paths, or
local usernames.

### notes

**Type:** string
**Required:** no
**Description:** A brief human-readable note about this publication, such as the
rationale for the selection of artifacts or the publication event that triggered
it. Must not contain private evidence details.

## Compatibility Reference: Sylus Scheduled Observation

The following describes how this contract maps to the Sylus Scheduled
Observation Foundation. This is a compatibility example, not a runtime
implementation.

Sylus produces a resident publication after its scheduled observation cycle
completes and the resulting evidence reaches `canonical` state. The publication
lists all canonical Sylus observation artifact references, carries the current
`distributionVersion`, and provides a `provenanceRef` pointing to the Sylus
evidence admission log. Deep Orbit receives this publication and derives a
presentation index from it.

```text
Canonical Sylus evidence
        ↓
Resident publication (Sylus)
        ↓
Presentation index
        ↓
Deep Orbit read
```

## Non-Goals

This contract does not define:

- a runtime publisher, scheduler, or ingestion adapter
- broker connectivity or live data feed integration
- the Canonical Evidence Contract (see `CANONICAL_EVIDENCE_CONTRACT.md`)
- the Presentation Index Contract (see `PRESENTATION_INDEX_CONTRACT.md`)
- the Deep Orbit Read Contract (see `DEEP_ORBIT_READ_CONTRACT.md`)
- Deep Orbit UI behavior
- paper or live trading
